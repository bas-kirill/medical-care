# Medical Service

## Rationale

```Medical Service``` интегрируется с системами пунктов вакцинации.

Сервис получает данные о вакцинации от всех пунктов вакцинации в формате CSV файлов. Строка каждого файла представляет собой:
* ФИО
* Паспорт
* Дата вакцинации
* Название вакцины
* Уникальный номер пункта вакцинации
* Название пункта вакцинации
* Адрес пункта

Сервис выполняет сохранение информации о вакцинации в базу данных посредством загрузки и обработки CSV файла:
* POST ```/process-file``` - multipart - загрузка реестра проведенных вакцинаций, его валидация, обработка и сохранение.

При загрузке файла необходимо проверить в ```Person Service``` валидность связки "ФИО, Паспорт". Для этого используется метод:
* GET ```/person/verify?name=...&passport=...```

Также сервис умеет предоставлять информацию о проведенной вакцинации для конкретного человека по данным его паспорта. Считается, что вакцина однокомпонентная и после одной вакцинации человек в безопасности
* GET ```/vaccination?document=...``` - получение подробной информации о вакцине по документу, удостоверяющему личность

В сервисе ```Medical Service``` периодически запускается планировщик, который обрабатывает все новые поступившие записи о вакцинации, и отправляет их в Kafka для последующей генерации QR кодов.

## Сущности:
* Пункт вакцинации: 
  * Уникальный номер;
  * Наименование;
  * Город;
  * Адрес;
* Вакцинация:
  * Уникальный номер;
  * Дата вакцинации;
  * ФИО пациента;
  * Документ удостоверяющий личность;
* Вакцина:
  * Название;


## API:
* POST ```/process-file``` (multipart): загрузка реестра проведенных вакцинаций, его валидация, обработка и сохранение;
* GET ```/vaccination?document=...```: получение подробной информации о вакцине по документу, удостоверяющему личность;


## Дополнительные требования:
* Сервис должен взаимодействовать со своей собственной БД
* При взаимодействии с Person service следует учесть вероятность его недоступности и использовать паттерны, обеспечивающие отказоустойчивость.
* Вести логи обработки реестра вакцинации
* Использовать трассировку запросов
* Файл с реестром о вакцинациях следует сохранить в репозитории, чтобы можно было запустить и проверить работу сервисов
* Формат сообщения в Kafka - Json. (также можно использовать Avro или Protobuf)
* Структура сообщения должна содержать минимум данных для удовлетворения бизнес требованиям системы.
* Событие вакцинации должно быть отправлено в Kafka хотя бы раз.


Сборка JAR: ```mvn clean compile assembly:single```

Стек микросервиса: Spring Boot, Spring Data, Spring Web, Spring OpenFeign
