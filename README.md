## Компоненты системы:
* ```Person service```: микросервис ответственный за работу с пользователями системы;
* ```Medical API service```: сервис для интеграции с пунктами вакцинации;
* ```QR-code storage service```: сервис для генерации и персистентного хранения QR кодов пользователей;
* ```API Gateway```: ответственнен за маршрутизацию запросов поступающих в систему
* Все сервисы зарегистрированы в service registry;

Для достижения целевого результата в рамках работы предстоит решить ряд задач.

## Задача 1. Создание person service.

Разработать сервис person-service. Сервис хранит личную информацию о гражданах: ФИО, паспортные данные,  другие документы, телефоны, адреса и пр.

По каждому человеку вносится информация о нем через сотрудников поддержки.

У одного гражданина могут быть:
* Несколько документов (паспорт, загранпаспорт, снилс, страховой полис)
* Несколько адресов проживания и один адрес регистрации
* Несколько контактов (телефоны, email и тп)

Сущности: 
* ```Person``` - основная сущность, на которую все завязано
* ```IdentityDocument``` - [OneToMany] разные документы, удостоверяющие личность, работа с enum
* ```Address``` - [ManyToMany]
* ```Contact``` - контакты [OneToMany] 


Что надо сделать:

* Релизовать доменную модель. Создать таблицы, сущности и закрепить связи между ними.
* Создать слой репозиториев для основных запросов к этим сущностям
* Создать сервисные методы, вызывающие репозитории
* Создать rest api исходя из требований. 
* Предусмотреть обработку исключений
* Настроить логирование запросов/ответов

Опционально:
* Добавить возможность применять миграции при изменении схемы данных с помощью liquibase / flyway / pgmigrate.

## Задача 2. Создание Medical service

Разработать сервис Medical API для интеграции системы с пунктами вакцинации.

Сервис получает данные от всех пунктов вакцинации в виде csv файла с информацией о том, кому, когда и какую вакцину поставили. От ковида, разумеется.

Сервис выполняет функционал сохранения информации о вакцинации посредством загрузки и обработки csv файла. Этот файл имеет следующую структуру:
* ФИО
* Паспорт
* Дата вакцинации
* Название вакцины
* Уникальный номер пункта вакцинации
* Название пункта вакцинации
* Адрес пункта


При загрузке файла необходимо проверить в Person service валидность связки фамилия+паспорт.  Для этого использовать метод, реализованный в предыдущем задании:

* GET ```/person/verify?name=...&passport=...``` проверить валидностть связки ФИО+паспорт.


Помимо этого сервис должен уметь предоставлять информацию о проведенной вакцинации для конкретного человека по данным его паспорта. При этом считается, что в нашем высокотехнологичном мире вакцина однокомпонентная и человек находится в безопасности после одной дозы.


Сущности:
* Пункт вакцинации - уникальный номер (сертификат гос. образца), наименование, город, адрес
* Вакцинация - уникальный номер (чтоб ничего не украли), дата чипирования, ФИО пациента, документ удостоверяющий личность
* Вакцина - название


API:
* POST ```/process-file - multipart``` - загрузка реестра проведенных вакцинаций, его валидация, обработка и сохранение. 
* GET ```/vaccination?document=...``` - получение подробной информации о вакцине по документу, удостоверяющему личность


Дополнительные требования:
* Сервис должен взаимодействовать со своей собственной БД
* При взаимодействии с Person service следует учесть вероятность его недоступности и использовать паттерны, обеспечивающие отказоустойчивость.
* Вести логи обработки реестра вакцинации
* Использовать трассировку запросов
* Файл с реестром о вакцинациях следует сохранить в репозитории, чтобы можно было запустить и проверить работу сервисов


## Задача 3. Отправка событий вакцинации:

В сервис Medical API добавить запуск планировщика, который обработает все новые поступившие записи о вакцинации, и отправит их в Kafka для последующей генерации QR кода.

Дополнительные требования:

* Формат сообщения в Kafka - Json. (также можно использовать Avro или Protobuf)
* Структура сообщения должна содержать минимум данных для удовлетворения бизнес требованиям системы.
* Событие вакцинации должно быть отправлено в Kafka хотя бы раз.

## Задача 4. Создание сервиса генерации QR кодов

Сервис генерирует и хранит сам QR-код и основную информацию о нем, которую вычитывает из сообщений Kafka. Вместо реального QR кода генерируем хэш по алгоритму MD5 от полученных из Kafka данных. Далее под QR кодом подразумеваем именно этот хэш. Сервис выставляет REST API для получения данных о QR коде пользователя и его валидации.

Требования к API:

* GET ```/qr/{passport}``` Метод, который выдает последний самый свежий код пользователю для последующего предъявления по его паспорту
* GET ```/qr/check?сode={code}``` Метод, который принимает QR-код и удостоверяет его подлинность и актуальность и, тем самым, подтверждает полномочия владельца.

Дополнительные требования:

* На каждое уникальное событие вакцинации должен быть создан ровно один QR код.
* Сервис обеспечивает персистентное хранение QR кодов в БД.
* Сервис зарегистрирован в service registry

## Задача 5. Создание API Gateway

API Gateway станет точкой входа в нашу систему. Все запросы пользователей пойдут через него. Он должен обеспечивать маршрутизацию запросов направляя их в целевые микросервисы и имплементировать дополнительные REST API методы.

Требования к API:

* ```/api/v1/person``` - вызовы должны проксироваться на микросервис Person.
* ```/api/v1/qr``` - проксируется на QR микросервис.
* ```/api/v1/medical``` - проксирует на medical.
* GET ```/api/v1/info/{passport}``` - метод агрегирующий информацию по клиенту от всех существующих микросвервисов по его паспорту.

Требования к микросервису:

* Обеспечивает маршрутизацию запросов, поступающих извне проксируя их на целевые микросервисы;
* Реализует метод, который вернет для пользователя краткую информацию о нем, информацию о его QR-коде и вакцинации. Для этого потребуется выполнить 3 вызова к существующим микросервисам, агрегировать полученный результат и вернуть его;
* Для реализации агрегирующего метода в микросервисе Person потребуется реализовать метод получения информации о клиенте по паспорту;

Опционально:
* Метод получения агрегированных данных реализовать в реактивном стиле.
